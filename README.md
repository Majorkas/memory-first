# Memory First

**Live site:** https://memory-first.onrender.com/

Memory First is a Django web application designed to support **patients** living with memory challenges and the **carers** who support them. The core feature is a daily **Memory Game** that helps patients practice recalling important people in their lives (e.g., family/friends), while providing carers dashboards and stats about engagement and accuracy.

This repo is a Django project with two primary apps:

- `user` — authentication, roles (carer/patient), dashboards, care relationships, profile management, family/friend management, reminder middleware
- `memory` — the Memory Game UI + API endpoints + attempt tracking

- `/api/family-memory/question/` (GET) — Get the next memory question

## Local development

### Prerequisites
- Python installed (version compatible with project)
- **uv** installed
- Node.js + npm (for Tailwind build)

### 1) Clone
```bash
git clone https://github.com/Majorkas/memory-first.git
cd memory-first
```

### 2) Create venv and install Python dependencies (uv)
```bash
uv venv
# macOS/Linux:
source .venv/bin/activate
# Windows (PowerShell):
# .\.venv\Scripts\Activate.ps1

uv sync
```

### 3) Install Node deps and build Tailwind CSS
This repo commits `memory_first/static/css/output.css`, but you should still build locally when changing styles.

```bash
npm ci --prefix memory_first
npm --prefix memory_first run build:css
```

### 4) Create `.env` in repo root
The project loads environment variables via `python-dotenv`. Create:

```bash
DJANGO_KEY="dev-secret-key-change-me"
DEBUG="TRUE"
DATABASE_URL="sqlite:///db.sqlite3"
```

Optional (for Cloudinary uploads in dev):
```bash
CLOUDINARY_CLOUD_NAME="..."
CLOUDINARY_API_KEY="..."
CLOUDINARY_API_SECRET="..."
```

### 5) Migrate and run
```bash
uv run python memory_first/manage.py migrate
uv run python memory_first/manage.py runserver
```

Open: http://127.0.0.1:8000/

## Deployment on Render (Native Environment)

This project is deployed on **Render** using a **Native Environment** and **Render Postgres**.

### Environment variables (Render)

Set these in your Render Web Service settings (Environment):

**Required**
- `DJANGO_KEY`  
  Django `SECRET_KEY` value (long random string).
- `DEBUG`  
  Set to `FALSE` in production.
- `DATABASE_URL`  
  Provided by Render Postgres (connect your DB to the service or copy the internal URL).

**Cloudinary (media uploads)**
- `CLOUDINARY_CLOUD_NAME`
- `CLOUDINARY_API_KEY`
- `CLOUDINARY_API_SECRET`

**Render Hostname (recommended)**
- `RENDER_EXTERNAL_HOSTNAME`  
  Render sets this in many setups. The settings use it to extend `ALLOWED_HOSTS`.

> Email: **Not configured**. On free Render, outbound email may not be available. Password reset emails and related flows should be considered non-functional unless you add an email provider.

### Build & start commands

You provided a single “full start command” that does install/build/migrate/seed/admin bootstrap. On Render, it’s more typical to split install/build into **Build Command** and keep **Start Command** as the actual server start, but you can keep your approach if you prefer.

#### Recommended Render setup (split build vs start)

**Build Command**
```bash
uv sync --frozen && npm ci --prefix memory_first && npm --prefix memory_first run build:css && uv run python memory_first/manage.py collectstatic --noinput
```

**Start Command**
```bash
uv run python memory_first/manage.py migrate && uv run python memory_first/manage.py startup_seed && uv run python memory_first/manage.py ensure_superuser && uv run gunicorn memory_first.wsgi:application
```


### Database & migrations

- Production requires `DATABASE_URL` (Render Postgres).
- The deploy commands run:
  - `python memory_first/manage.py migrate`

### Static files

Production uses:
- `collectstatic` to write to `STATIC_ROOT`
- WhiteNoise middleware/storage for serving static assets

The deploy commands run:
- `python memory_first/manage.py collectstatic --noinput`

### Media files (Cloudinary)

Media storage is configured to use Cloudinary (django-cloudinary-storage). Ensure the Cloudinary env vars are present or uploads will fail.

## Architecture

### User roles and permissions
- There are two roles:
  - **Carer**
  - **Patient**
- Patients **cannot self-signup**.
  - Patient accounts are expected to be created/managed by carers inside the application.
  - There is **no invite-by-email** flow.

### High-level request flow (Memory Game)

1. Patient navigates to `/game/`
2. Front-end JS calls `GET /api/family-memory/question/` to retrieve a question + person image
3. Patient submits an answer to `POST /api/family-memory/submit/`
4. Server records a `MemoryGameAttempt` and returns correctness
5. Carer dashboard aggregates attempts for stats and reminders

### Reminder behavior
A middleware checks whether a patient has played recently and shows a reminder message if not, with a “snooze” endpoint that uses session data.

## Apps (in-depth)

## `user` app

### Purpose
The `user` app owns:
- Custom user model and role handling
- Carer/patient profiles
- Linking carers to patients
- Managing a patient’s “Family & Friends” (the people used by the Memory Game)
- Carer dashboards and patient detail pages
- Reminder middleware + snooze endpoint

### Key concepts (models)
This section describes the intent of the major data models used by the app:

- **Custom User**: a single user table with a `user_type` (carer/patient)
- **PatientProfile / CarerProfile**: role-specific profile data (pictures, demographics, etc.)
- **PatientCarerRelationship**: links carers to patients
- **FamilyFriend**: the people a patient should remember (image, name, relationship, emergency contact)

### Signals
User creation triggers automatic creation of the correct profile model.

### Middleware
A “memory game reminder” middleware warns patients if they have not played in the last 24 hours and supports a session-based snooze.

### Views
The dashboards are role-aware:
- Carers can create/manage patient accounts, manage patient family/friends, and view stats.
- Patients can view their own relevant info and play the game.

---

## `memory` app

### Purpose
The `memory` app provides:
- The Memory Game page
- Two API endpoints:
  - get a question
  - submit an answer
- An attempt model to persist play history for stats/reminders

### Attempt tracking
Each question answered creates a `MemoryGameAttempt` record capturing:
- patient
- which person (FamilyFriend)
- question type (name/relationship)
- expected answer vs user answer
- correctness
- timestamp

## API documentation

Base path: your deployed site (e.g., `https://memory-first.onrender.com/`)

### GET `api/family-memory/question/`

**Auth:** must be logged in as a **patient**.

**Purpose:** returns the next memory question (random person + question type), using the session to avoid repeating the same person within a game.

**Response (shape)**
- question prompt
- question type
- person info (including image URL)

### POST `api/family-memory/submit/`

**Auth:** must be logged in as a **patient**.

**Body**
```json
{ "answer": "..." }
```

**Behavior**
- Compares normalized answer to expected answer stored in session
- Records a `MemoryGameAttempt`
- Returns correctness and expected answer when incorrect

## Troubleshooting (Render-focused)

### 1) Build fails on Node/Tailwind
Symptoms:
- Tailwind build command fails
Fix:
- Ensure Render has a Node runtime available in the Native Environment.
- Confirm `npm ci --prefix memory_first` works locally.
- Verify the `memory_first/package.json` scripts include `build:css`.

### 2) `DATABASE_URL` missing
Symptoms:
- Deploy crashes on startup, database config errors
Fix:
- Ensure Render Postgres is connected and `DATABASE_URL` is set in environment variables.

### 3) Static files missing (CSS not loading)
Symptoms:
- Site renders unstyled
Fix:
- Ensure `collectstatic` runs during build/deploy.
- Confirm WhiteNoise middleware is enabled and `STATIC_ROOT` exists after collectstatic.

### 4) Cloudinary uploads failing
Symptoms:
- Upload errors, broken media links
Fix:
- Confirm `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET` are set.
- Verify Cloudinary credentials have permission to upload.

### 5) Password reset / email flows not working
Expected:
- Email is **not configured** for free Render setup.
Fix:
- Add an SMTP provider and configure Django email settings if you need password reset emails.
